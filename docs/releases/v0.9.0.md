# 苏小有 v0.9.0 候选发布

状态：`v0.9.0-rc.1` 候选发布
目标版本：`0.9.0`
发布主题：可终止执行、安全默认值、可恢复升级与可审计安装包

## 成熟度结论

当前实现达到 RC 成熟度，但还不能判定为稳定版成熟。`v0.9.0-rc.1` 只会在同一 tag 的所有
原生 runner 完成最终安装、启动、退出、沙箱和升级/恢复门禁后发布；公开 Release 及其工作流
记录是候选产物通过该门禁的证据。当前只有 Linux Bubblewrap 同时满足隔离与后代进程
强制回收；macOS 与 Windows 均关闭 Bash/Python 执行。若产品要求三平台都能执行命令，
则该要求仍未完成。

## 执行沙箱

- Linux 通过 Bubblewrap 创建独立的挂载、网络和进程边界。DEB 与 RPM 分别声明
  `bubblewrap` 生产依赖；程序找不到 `bwrap` 时拒绝执行，绝不降级到普通子进程。
- macOS Seatbelt 可以限制文件系统、网络与环境，但公开、无需额外授权的进程控制接口不能
  保证回收一个在沙箱内 `setsid` 后脱离的后代；仅有资源隔离不足以满足“可终止”要求。因此
  `bash` 与 `code_execute` 在 rc.1 fail-closed。重新开放前需增加能覆盖完整后代生命周期的
  强边界，并在最终 `.app` 内证明超时后没有残留进程。
- Windows Job Object 可以负责超时、停止和应用退出时回收进程树，但没有提供工作区文件系统
  或网络隔离，所以同样 fail-closed。后续只有引入 AppContainer、受限令牌加文件系统 ACL，
  或等价隔离后才能重新开放。

## 权限和首次启动

- 默认工作模式为 Ask，文件修改与命令执行均不预先批准。
- 首次启动必须选择有效工作区后才能完成引导；迁移后的配置若没有有效工作区，同样重新要求
  选择，避免把 `.` 或无限制路径当作历史默认值。
- 非交互任务没有用户可以回答权限问题，因此遇到 `ask` 必须终止并返回可诊断错误。
- 子任务收到父任务 workspace、permission mode 和批准上下文的不可扩张副本；子任务请求的
  工作区与父任务不同或缺少边界时拒绝启动。

## Provider 与最终 bundle

- Anthropic 使用官方 `anthropic` SDK，Gemini 使用官方 `google-genai` SDK；二者都进入锁定的
  Python 生产依赖和 PyInstaller 收集规则。
- 适配器覆盖文本、图片、system、工具定义、工具结果、JSON schema、usage 和
  stop/tool-call/length 完成原因，并使用 SDK 形状的假事件执行离线单元回归。
- Anthropic extended thinking 与 Gemini thinking 的工具往返需要原样保存签名块。当前通用消息
  历史没有该能力，因此两个原生适配器都标记 `reasoning=false`，且不发送 thinking 配置；为
  兼容前端遗留偏好，即使请求仍带 `reasoning=true` 也会安全忽略，不以丢弃签名的半实现
  冒充生产支持。
- 冻结后端提供网络无关的 provider self-test：使用假凭据构造两个官方客户端并核对实际
  类型。bundle verifier 在启动 HTTP smoke 前先执行 provider 与沙箱 self-test：Linux 必须
  给出四项隔离证明，macOS/Windows 必须报告 disabled，避免源码可用但最终包遗漏动态模块，
  也避免非 Linux 产物意外退回无隔离执行。

## 数据、凭据和回滚

- v0.9.0 使用显式 `0005_v090_release_boundary` Alembic revision。即使本轮没有表结构变化，
  也不再靠版本猜测判断数据库来自旧版还是未知新版。
- 升级前使用 SQLite online backup；备份清单记录 SHA-256、大小、源/目标 revision 和时间。
  离线恢复先验证清单、校验和、`quick_check` 与 revision，再生成临时数据库并原子替换；替换
  前额外保存当前数据库，不执行正式库上的原地 downgrade。
- Provider、OAuth、MCP 与 Channels 的敏感字段只在配置中保存不透明引用。生产环境优先使用
  macOS Keychain、Windows Credential Manager 或 Linux Secret Service；原生服务不可用时的
  文件回退使用 `0600` 权限和原子替换。
- 操作步骤见[数据库升级与离线恢复](../database-recovery.md)。

## Remote 与 Channels

当前 Remote bearer 和消息渠道都会把外部输入带入能访问主机工具的 Agent，尚未完成隔离、
来源授权和无人值守审批设计。因此 rc.1 用代码常量关闭两者：不挂载可用管理能力、不启动
Cloudflare/manual tunnel、不启动 channel consumer 或 Agent adapter。该开关不能被遗留 `.env`
重新打开。未来重新设计必须使用能力裁剪的 Agent、独立 workspace、不可交互的 deny 默认值，
并重新完成威胁模型与跨来源回归。

## CI 与原生安装包门禁

普通 CI 必须完成：

- 带哈希 Python 锁安装、Python 漏洞审计和五个目标平台的 wheel 可用性检查；
- 仓库根、frontend、desktop-tauri 三份 JavaScript 生产依赖审计；
- production frontend、完整后端、Rust test/clippy/release build 与 Rust 漏洞审计；
- Playwright Chromium 核心交互；
- Linux 冻结后端与 Node runtime 打包、Tauri DEB 构建、包元数据检查、解包后的 bundle smoke、
  `dpkg -i` 真实安装和 purge。

Release tag 还必须在原生 Windows x64、macOS arm64、`macos-15-intel`、Linux x64 与 Linux
ARM64 runner 生成同一提交的七个安装包。Intel runner 必须重建 Python 后端与 Rust/Tauri，
验证最终 `.app` 和挂载/复制后的 DMG 应用的 Mach-O 架构、macOS 11.0 最低版本、签名状态、
内置 runtime、启动、正常退出与无孤儿进程；不接受 universal2 壳包内嵌错误架构后端。

### Rust 审计已知项

本地 `cargo audit` 已把 `quick-xml`、`rustls-webpki` 与 `quinn-proto` 的实际 vulnerability 更新到
修复版本，当前报告 `0 vulnerabilities`。仍有 19 条不会触发默认非零退出码的 warning，不能
把它们写成“零风险”：

- 17 条 `unmaintained` informational：GTK3/gtk-rs 链的 `atk*`、`gdk*`、`gtk*`，以及
  `fxhash`、`proc-macro-error` 和五个 `unic-*` crate，主要来自 Tauri Linux WebKit/HTML 依赖链；
- 2 条 `unsound` RustSec warning：`glib 0.18.5` 与 `rand 0.7.3`。前者来自 Tauri 的 GTK3
  链；后者只经 PHF 生成器进入构建期依赖路径。它们不是本轮被忽略的 vulnerability，但仍是
  需要跟随上游升级的安全债务。

RC 说明必须保留这一分类。CI 的 `cargo audit` 会阻断 vulnerability；上述 warning 暂作为显式
风险接受，不使用 ignore 列表隐藏，也不得对外概括为“Rust 依赖无警告”。

## RC 发布阻断项

1. 所有普通 CI 门禁通过，包括真实 Linux DEB 安装事务和最终 bundle 双 self-test。
2. 用 v0.8.3 实际数据库完成一次升级、备份清单核对、离线恢复和重新启动演练。
3. 同一 `v0.9.0-rc.1` tag 的七个原生安装包全部通过安装与桌面生命周期验证。
4. 对 macOS 与 Windows 安装包确认命令工具确实 fail-closed；Windows 仍要验证桌面后端生命周期
   使用 Job Object 回收后代，但不能把这一点写成命令沙箱已经可用。
5. 审核漏洞报告；任何 vulnerability 必须修复，不能用 ignore 让门禁变绿。非漏洞 advisory 也要
   记录并逐项评估，不把 `cargo audit` 的零退出码解释为零警告。
6. macOS RC 若没有 Developer ID 和公证凭据，文件名、Release 说明与 manifest 必须继续明确
   标注 ad-hoc 和未公证；稳定版签名门禁不放宽。
7. Release workflow 只创建草稿。发布负责人必须在工作流全部成功后核对恰好七个安装包、
   `CHECKSUMS.md` 与 `release-manifest.json`（共九个资产），抽验下载后的校验和及最终应用，
   再把草稿提升为公开 prerelease，并确认匿名访问发布页和九个资产均成功。

## 候选版边界

- Release 页面和同一 tag 的工作流记录才是 Intel、Windows 与 Linux ARM64 产物通过验证的
  证据；不能用工作流配置或本地预检代替。
- macOS 与 Windows 的命令执行能力没有完成；rc.1 的成熟结论只建立在“安全关闭”而非
  “三平台功能对齐”上。
- 本地 Apple Silicon 构建只能作为 arm64 预检，不能替代原生 Intel runner 或正式签名/公证结果。
