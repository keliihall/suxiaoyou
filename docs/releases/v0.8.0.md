# 苏小有 v0.8.0 发布计划

状态：实施中
目标版本：`0.8.0`
发布主题：稳定性、长任务可控性与文件体验

## 发布原则

v0.8.0 必须同时满足以下四个结果：

1. 数据不丢失，上传文件、生成文件和数据库升级均可恢复。
2. 操作不重复，同一会话不会并发执行多个用户回合，网络重试不会重复产生副作用。
3. 长任务状态真实，用户可以确认、继续输入、排队、引导、停止并在重启后恢复待办输入。
4. 安装包可验证，Windows、macOS 和 Linux 的正式发布产物通过真实安装与启动门禁。

## P0 发布阻断项

### 1. 请求与执行幂等

- 默认仅允许 GET/HEAD 自动重试。
- 可能触发 Agent 执行的首条 prompt，以及运行中的排队/引导请求，必须使用客户端请求 ID；重复请求返回同一结果。
- 同一会话任意时刻最多存在一个活跃 generation。
- 桌面端与移动端同时提交时，由服务端原子排序且不重复执行。

验收：分别模拟首条 prompt 与排队/引导请求在服务端已提交但响应中断，Agent 执行、工具副作用和费用均只发生一次。

### 2. 附件与输出文件安全

- 相同内容跨会话共享时，删除一个会话不得破坏其他引用。
- 文件编辑、取消队列和删除会话只减少引用；最后一个引用消失后才能回收物理文件。
- 未选择项目时，为会话创建稳定的托管输入、输出和临时目录。
- 上传文件的父目录只作为输入来源，绝不自动成为输出目录。
- 只有用户明确指定时，才允许保存到源文件旁或覆盖源文件。

验收：两个会话引用同一文件、删除任一会话后另一会话仍可读取；无项目会话不会在源文件目录生成任何新文件。

### 3. 正式数据库迁移与恢复

- 建立 Alembic 基线和 `0.8.0` revision。
- 升级前使用 SQLite online backup，并执行 `quick_check`。
- 迁移失败时保留原库和备份，应用进入可理解的恢复界面。
- 迁移重复执行必须幂等。

验收：真实 `0.7.3` 数据 fixture 可升级；迁移中断后原数据仍可打开。

### 4. 长任务交互状态机

统一状态：

`排队中 → 思考中 → 调用工具 → 等待确认 → 正在提交 → 已确认 → 正在继续 → 完成/失败/超时/已停止`

- 确认、问题和计划审批必须持久化并具有明确回执。
- `/chat/respond` 对不存在、过期、已处理和冲突回答返回明确语义。
- 传输心跳与业务进展分开；非等待用户状态长时间无业务事件才显示“可能卡住”。
- 时长使用本地化的秒、分秒、小时分秒，不显示裸秒总数。

验收：确认成功后 2 秒内显示回执和后续工具状态；错误或重连时确认卡不会静默消失。

### 5. 运行中排队与引导

- 运行时输入框、粘贴和附件保持可用。
- 默认“排队”：当前回合正常结束后 FIFO 执行。
- 可选“引导当前任务”：在模型或工具批次结束后的下一安全边界恰好应用一次。
- 停止、失败或崩溃默认暂停队列；不得自动重放可能产生副作用的任务。
- 队列及附件在刷新、切换会话、重启应用和多端访问后保持一致。

验收：10 分钟长工具期间加入 3 条消息，按顺序各执行一次；引导内容在下一安全边界可见并仅应用一次。

### 6. 工具停止、进度与异常恢复

- 停止必须终止当前工具及其子进程树，而不只是关闭前端 SSE。
- 退出应用时运行工具标记为 interrupted，不能永久保留 running。
- 长命令提供业务进度或最后更新时间；超时、重试和步骤上限可见。

验收：停止长命令后不存在孤儿进程，数据库和 UI 均显示已停止。

### 7. 服务单例与自动化正确性

- 完成 ADR-0008，禁止继续读取未初始化的 `app.state.index_manager` 或 `app.state.stream_manager`。
- 自动化启动先处理漏跑任务再重算下一时间。
- 循环执行失败不得标记成功；关机和重启能正确落库与恢复。

验收：FTS、远程任务状态和循环输出可用；漏跑恰好补一次；失败状态准确。

### 8. 文件操作体验

- 文件卡同时提供可见菜单和右键菜单。
- 本地桌面支持默认应用打开、文件管理器定位、复制路径、另存为/下载。
- 远程端只提供预览和下载，不暴露本机路径或启动系统应用。
- 文件不存在、过大或系统打开失败时显示明确错误。

### 9. 稳定的一级导航入口

- “新建项目”和“新对话”不依赖列表是否已有数据，始终可访问。
- 覆盖零项目、零会话、搜索、按时间组织、仅置顶和列表错误状态。
- Windows 10/11 的中文、空格和盘符路径必须通过文件夹选择回归。

### 10. 发布门禁

- PR：后端、前端 unit/type/lint、Rust test/clippy、Chromium 核心 Playwright。
- RC：Windows NSIS、macOS arm64/x64、Linux deb/rpm 的安装后启动、后端 ready、退出无孤儿进程。
- 升级：`0.7.3 → 0.8.0` 数据与配置保留。
- 正式产物：版本一致、签名/公证状态明确、校验和和 release notes 完整。

## P1（不阻断 v0.8.0）

- 完整枚举跨平台“打开方式”应用列表。
- 队列自由拖拽重排和跨会话全局优先级。
- 在模型 token 流中即时注入内容。
- 完整应用内自动更新；v0.8.0 只要求可信版本发现与下载入口。
- 全面视觉重构。
- 大文件“另存为”改为原生按路径复制或流式传输，避免 base64 与桌面 IPC 的内存放大；v0.8.0 先限制单次内容大小并保证覆盖写入原子性。
- 上传孤儿文件改为运行时周期回收；v0.8.0 在启动维护阶段执行引用感知清理。

## 实施顺序

1. 数据迁移、附件引用、写入边界、请求幂等。
2. 单会话协调器、确认回执、持久化队列和安全边界引导。
3. 工具停止与恢复、服务单例、自动化正确性。
4. 文件菜单、时间文案、稳定导航入口和诊断信息。
5. 单元/集成/UI/Rust 门禁与三平台 RC。
6. 同步版本、生成 release notes、合并、打 `v0.8.0` tag、验证正式 Release。

## 发布完成证据

只有以下证据全部存在，才可宣告 v0.8.0 完成：

- 本文所有 P0 验收项对应的自动化测试通过。
- 工作树干净，版本元数据全部为 `0.8.0`。
- 主分支 CI 通过，正式 tag 的全部 release jobs 通过。
- GitHub Release 非草稿，包含 Windows、macOS 双架构、Linux deb/rpm、`CHECKSUMS.md` 与仅用于版本发现/手动下载的 `release-manifest.json`。
- 下载的正式产物重新校验 SHA-256；macOS 签名/公证、Windows 安装、Linux 包内容均有验证记录。
