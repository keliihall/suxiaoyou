# v1.0 RC 验收计分卡

`scripts/verify-v1-rc-scorecard.mjs` 将 v1.0 RC 的外部门禁变成机器可判定的证据。模板
`docs/releases/v1.0-rc-scorecard.template.json` 只描述字段，不是完成证据；不要把模板中的空值或
默认值当作测试结果。

## RC 门禁

- 七个候选安装包来自同一个 `v1.0.0-rc.N` tag 和 commit，每包必须提供安装包与
  生命周期报告 SHA-256，并完成安装、启动、退出和无孤儿进程验证；
- macOS RC 必须是 Developer ID 包，或明确标注 `rc-adhoc` 并验证未签名/未公证信任边界；
- Windows x64、macOS arm64/x64、Linux x64/ARM64 的冻结后端必须通过 DOCX/XLSX/PPTX
  创建、有限编辑、独立重开、原子安装和写前版本快照；每个原生报告必须与发布
  tag/commit 一致并提供 SHA-256；
- 至少 20 名受控用户、200 个核心任务、连续 7×24 小时，且五类核心工作流均有覆盖；
- 总任务成功率不低于 90%，所有输出文件均重开验证，不可恢复数据丢失为 0；
- 后端、前端、Playwright、Rust、迁移、安全边界和供应链门禁通过，未关闭 P0/P1 为 0；
- 腾讯文档和图片 Provider 的 mock/合同测试通过。真实账号 E2E 在 RC 只能标记为
  `pending_credentials` 或 `passed`，GA 模式必须为 `passed`。

Beta 数字不得直接手填。每个测试任务只记录匿名参与者 SHA-256 token、核心工作流、时间、成功、
输出重验、不可恢复数据损失计数和该任务本地证据的 SHA-256；不得采集提示词、文件路径或真实身份。
将这些记录保存为 JSONL 后生成绑定候选 tag/commit 的汇总：

```json
{"schema_version":1,"task_id":"beta-task-0001","participant_token":"1111111111111111111111111111111111111111111111111111111111111111","release_tag":"v1.0.0-rc.2","release_commit":"aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa","workflow":"office_create_edit","started_at":"2026-07-14T01:00:00.000Z","ended_at":"2026-07-14T01:05:00.000Z","succeeded":true,"output_task":true,"output_revalidated":true,"unrecoverable_data_loss":0,"evidence_sha256":"bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb"}
```

`participant_token` 应由每名参与者的随机 Beta 标识与本轮随机盐生成，不能直接散列姓名、邮箱或
设备序列号；`evidence_sha256` 对应操作者本地保存的单任务验收记录。工具拒绝重复任务 ID、重复
任务证据、额外隐私字段、候选版本不一致、非规范 UTC 时间以及符号链接输入/输出。

```bash
npm run summarize:v1-beta-evidence -- \
  --input path/to/anonymous-beta-tasks.jsonl \
  --output path/to/BETA-EVIDENCE.json \
  --release-tag v1.0.0-rc.2 \
  --release-commit <40-char-commit>
```

把 `BETA-EVIDENCE.json` 的 `beta` 原样复制到计分卡，并填写 `beta_evidence` 的 summary/source
SHA-256、tag 和 commit。随后必须执行伴随门禁，防止数字、候选版本或原始证据摘要被替换：

```bash
npm run verify:v1-scorecard-beta-link -- \
  --scorecard path/to/v1.0-rc-scorecard.json \
  --summary path/to/BETA-EVIDENCE.json
```

GA 可以复用 RC Beta，但只能在 `v1.0.0` 与该 `v1.0.0-rc.N` 指向完全相同 commit 时复用；
`beta_evidence.release_tag` 仍保留实际测试过的 RC tag。不得把 RC 原始记录改写成 GA tag，也不得把
另一个提交上的 Beta 结果拼接进 GA 计分卡。

## 使用方式

将模板复制到证据目录并填入真实结果，然后运行：

```bash
node scripts/verify-v1-rc-scorecard.mjs path/to/v1.0-rc-scorecard.json
```

腾讯文档和图片 Provider 不能只填字符串。将对应 tag 工作流产出的
`INTEGRATION-CONTRACTS.json`（稳定 tag 会包含两项真实 E2E 的 `ga` summary）SHA-256、commit
和 tag 填入 `integration_evidence`，再运行伴随门禁：

```bash
npm run verify:v1-scorecard-integration-link -- \
  --mode rc \
  --scorecard path/to/v1.0-rc-scorecard.json \
  --summary path/to/INTEGRATION-CONTRACTS.json
```

该命令会拒绝 summary hash、release commit、release tag、合同状态不匹配；GA 还会拒绝任何未
标记 `passed` 或 summary 中缺少真实腾讯文档/图片 Provider gate 的结果。主计分卡、Beta
伴随门禁与集成证据伴随门禁必须同时通过。

稳定版完成审计使用更严格的 GA 模式：

```bash
node scripts/verify-v1-rc-scorecard.mjs path/to/v1.0-ga-scorecard.json --ga
npm run verify:v1-scorecard-integration-link -- \
  --mode ga \
  --scorecard path/to/v1.0-ga-scorecard.json \
  --summary path/to/INTEGRATION-EVIDENCE.json
```

任何缺失、无效计数、短于七天、低于成功率、未重验输出、数据丢失、缺包、未签名稳定 macOS 包
或未完成真实集成都会以非零状态退出。计分卡只汇总证据；安装日志、测试报告、原始任务记录和
缺陷清单仍应作为对应字段的可复核附件保存。

发布工作流会在五个原生 runner 上通过冻结 PyInstaller 后端执行合同，并把
`OFFICE-COMPATIBILITY.json` 与安装包一起附到候选发布。计分卡的 `office_compatibility`
应来自该文件，不应手工改写为布尔值。对 annotated tag，所有证据统一绑定
`HEAD^{commit}` 解引用后的提交，不把 tag object SHA 当作源代码提交。
